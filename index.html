🤔 Kubernetes là gì và tại sao cần?

Kubernetes là nền tảng mã nguồn mở để triển khai, vận hành và quản lý các ứng dụng containerized. Nếu Docker giúp chạy một container, Kubernetes quản lý rất nhiều container trên nhiều máy chủ.

Trong thực tế, một ứng dụng web thường gồm: Frontend, Backend API, Database, Cache, Message Queue, Worker, Monitoring Agent... Nếu chạy thủ công sẽ gặp ngay các vấn đề:

  • Container chết — ai khởi động lại?
  • Server hỏng — workload chuyển sang đâu?
  • Cần nhiều bản sao — làm thế nào?
  • Rolling update, service discovery, giới hạn tài nguyên?
ℹ️
Kubernetes sinh ra để giải quyết tất cả những bài toán đó — theo mô hình desired state: bạn khai báo trạng thái mong muốn, Kubernetes tự đưa cluster về đúng trạng thái đó.

🗺️ Kiến trúc Cluster tổng quan

Một Kubernetes cluster gồm hai nhóm thành phần: Control Plane (bộ não) và Worker Nodes (nơi chạy workload).

Kubernetes Cluster
│
├── Control Plane (master-01)
│   ├── kube-apiserver     ← Cổng giao tiếp trung tâm
│   ├── etcd              ← Database lưu trạng thái cluster
│   ├── kube-scheduler    ← Chọn node cho Pod
│   └── kube-controller-manager  ← Vòng lặp điều khiển
│
└── Worker Nodes (worker-01, worker-02)
    ├── kubelet           ← Node agent, chạy Pod
    ├── kube-proxy        ← Quản lý network rules
    ├── container runtime ← containerd
    └── CNI Plugin        ← Calico (network cho Pod)

⚡ Các thành phần Control Plane

Control Plane
kube-apiserver
Cổng giao tiếp trung tâm. Mọi request đều đi qua đây. Xác thực, phân quyền và ghi vào etcd.
Control Plane
etcd
Key-value store. Lưu toàn bộ trạng thái cluster. Mất etcd = mất cluster state.
Control Plane
kube-scheduler
Quan sát Pod chưa có node. Chọn node phù hợp dựa trên tài nguyên và constraints.
Control Plane
controller-manager
Chạy các controller vòng lặp: ReplicaSet, Node, Job... Đảm bảo desired state.
Worker Node
kubelet
Agent trên mỗi node. Nhận PodSpec, giao tiếp với container runtime để chạy Pod.
Worker Node
kube-proxy
Duy trì network rules trên node. Cho phép giao tiếp Pod qua Service.
Add-on
CoreDNS
DNS nội bộ cho cluster. Resolve tên Service thành ClusterIP.
Add-on
CNI (Calico)
Cấp IP cho Pod. Định tuyến traffic giữa các Pod trên các node khác nhau.

🔄 Luồng tạo một Pod

Khi bạn chạy kubectl apply -f pod.yaml, đây là những gì xảy ra:

1. kubectl → HTTP request → kube-apiserver
   └── Xác thực, phân quyền, validate object

2. kube-apiserver → Ghi Pod (Pending) → etcd

3. kube-scheduler → Phát hiện Pod chưa có node
   └── Chọn node phù hợp (worker-01)
   └── Ghi node binding → etcd qua API Server

4. kubelet trên worker-01 → Phát hiện Pod được gán
   └── Gọi container runtime (containerd)
   └── Pull image nếu chưa có
   └── Tạo và chạy container

5. kubelet → Báo cáo trạng thái → kube-apiserver
   └── Pod status: Running
⚠️
Kubectl không nói chuyện trực tiếp với kubelet hay containerd. Tất cả đều đi qua kube-apiserver. Đây là nguyên tắc cốt lõi của Kubernetes.

📝 Static Pod trong kubeadm

Trong mô hình kubeadm, các thành phần Control Plane tự chạy dưới dạng Static Pod. Kubelet đọc manifest từ thư mục cố định và chạy các Pod này mà không cần qua kube-apiserver.

bash 
# Xem các Static Pod manifest trên master node
ls /etc/kubernetes/manifests/
# kube-apiserver.yaml
# etcd.yaml
# kube-controller-manager.yaml
# kube-scheduler.yaml

# Xem các Pod hệ thống đang chạy
kubectl get pods -n kube-system

# Xem các node trong cluster
kubectl get nodes -o wide

🎯 Mục tiêu buổi học

Hiểu các workload resource chính của Kubernetes và khi nào dùng loại nào. Thực hành tạo, scale, rollout và kiểm tra từng loại.

🧠 Desired State và Actual State

Tư duy quan trọng nhất trong Kubernetes: bạn khai báo trạng thái mong muốn, controller tự đưa cluster về đúng trạng thái đó.

Desired State  →  "Tôi muốn 3 Pod nginx"
Actual State   →  "Hiện có 2 Pod đang chạy"
Controller     →  Tạo thêm 1 Pod để khớp desired state

Nếu một Pod chết:
Actual State   →  "Chỉ còn 2 Pod"
ReplicaSet     →  Tự tạo lại Pod thứ 3

🏷️ Label và Selector

Label là key-value gắn vào object. Selector dùng để lọc object theo label. Đây là cơ chế kết nối giữa các resource trong Kubernetes.

yaml
# Pod có label app=nginx, env=production
metadata:
  labels:
    app: nginx
    env: production

# Deployment selector: chọn tất cả Pod có app=nginx
selector:
  matchLabels:
    app: nginx

📊 So sánh các Workload Resource

ResourceDùng khi nàoTự restart?
PodTest, debug đơn giản❌
ReplicaSetDuy trì N bản sao Pod (thường dùng qua Deployment)✅
DeploymentỨng dụng stateless, cần rolling update✅
DaemonSetChạy 1 Pod trên mỗi node (logging, monitoring)✅
StatefulSetỨng dụng cần định danh ổn định (DB, Kafka)✅
JobTác vụ chạy một lần đến khi hoàn tấtOn failure
CronJobTác vụ chạy theo lịch (backup, report)On schedule

🔁 DaemonSet — Chạy trên mọi node

DaemonSet đảm bảo mỗi node (hoặc nhóm node) chạy đúng 1 Pod. Thường dùng cho log collector, monitoring agent, network plugin.

yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-logger
spec:
  selector:
    matchLabels:
      app: node-logger
  template:
    metadata:
      labels:
        app: node-logger
    spec:
      containers:
        - name: logger
          image: busybox:1.36

💼 StatefulSet — Định danh ổn định

StatefulSet cần khi ứng dụng cần: tên Pod ổn định (pod-0, pod-1, pod-2), storage riêng cho từng Pod, khởi động theo thứ tự.

ℹ️
StatefulSet thường dùng với Headless Service để các Pod có thể resolve DNS lẫn nhau: pod-0.my-service.default.svc.cluster.local

🎯 Mục tiêu buổi học

Hiểu Kubernetes networking model, tại sao Pod IP không đủ ổn định, và cách Service giải quyết bài toán service discovery.

⚠️
Câu hỏi trọng tâm: Khi Pod có thể bị xóa, tạo mới và đổi IP liên tục — làm sao ứng dụng tìm thấy nhau? → Service.

🌐 Ba lớp mạng trong Kubernetes

Layer 1
Node IP
IP thật của VM/server. Ổn định, có thể truy cập từ ngoài. Ví dụ: 192.168.1.10
Layer 2
Pod IP
IP do CNI cấp, nằm trong Pod CIDR (192.168.x.x). Thay đổi khi Pod restart. KHÔNG ổn định.
Layer 3
Service (ClusterIP)
IP ảo ổn định. Tồn tại suốt vòng đời Service. Đây là địa chỉ để ứng dụng phụ thuộc.

🔌 Phân biệt các loại Port

PortNằm ở đâuÝ nghĩa
containerPortPod specPort container lắng nghe (khai báo, không bind)
targetPortService specPort thực tế traffic được forward đến container
portService specPort Service lắng nghe (ClusterIP:port)
nodePortService specPort trên Node IP để truy cập từ ngoài (30000–32767)

🔵 ClusterIP Service

Loại Service mặc định. Chỉ truy cập được từ trong cluster. Dùng cho internal communication giữa các microservices.

yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-clusterip
spec:
  type: ClusterIP
  selector:
    app: backend       # Chọn Pod có label này
  ports:
    - port: 80            # ClusterIP:80
      targetPort: 8080    # → container port 8080
bash — kiểm tra
kubectl get service backend-clusterip
kubectl get endpointslices -l kubernetes.io/service-name=backend-clusterip

# Kiểm tra DNS từ trong cluster
kubectl run test --image=busybox:1.36 --rm -it -- \
  nslookup backend-clusterip.default.svc.cluster.local

🟢 NodePort Service

Mở một port trên mọi Node IP để truy cập từ bên ngoài cluster. Thích hợp cho lab và testing.

yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-nodeport
spec:
  type: NodePort
  selector:
    app: backend
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080    # Truy cập: NodeIP:30080
Từ ngoài cluster:
http://192.168.1.10:30080  →  kube-proxy  →  Pod:8080
http://192.168.1.11:30080  →  kube-proxy  →  Pod:8080

(cả 2 node IP đều forward được)

⭕ Headless Service

Service không có ClusterIP (clusterIP: None). DNS trả về danh sách IP của từng Pod thay vì một IP ảo. Dùng với StatefulSet để các Pod resolve được lẫn nhau.

yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres-headless
spec:
  clusterIP: None   # ← Đây là Headless Service
  selector:
    app: postgres
  ports:
    - port: 5432
💡
Với Headless Service, DNS resolve: postgres-0.postgres-headless.default.svc.cluster.local → trực tiếp IP của Pod postgres-0.

🏗️ Buổi 01 — Kiến trúc
0/5 đúng
CÂU 01
Trong Kubernetes, thành phần nào là cổng giao tiếp trung tâm của cluster?
kube-apiserver expose Kubernetes API và là điểm giao tiếp trung tâm giữa người dùng, công cụ quản trị và các thành phần nội bộ. Tất cả request đều đi qua đây.
CÂU 02
Thành phần nào lưu trạng thái của Kubernetes cluster?
etcd là key-value store dùng để lưu dữ liệu của Kubernetes API Server. Mất etcd = mất toàn bộ trạng thái cluster.
CÂU 03
Khi một Pod mới được tạo nhưng chưa được gán vào node, thành phần nào sẽ chọn node?
kube-scheduler quan sát các Pod chưa được gán node (nodeName trống) và chọn node phù hợp dựa trên tài nguyên, constraints và policies.
CÂU 04
Trong mô hình kubeadm, các manifest control plane thường nằm ở đâu?
kubeadm đặt các Static Pod manifest của control plane component trong /etc/kubernetes/manifests/. Kubelet đọc và chạy các Pod này trực tiếp.
CÂU 05
Pod là gì trong Kubernetes?
Pod là đơn vị deployable nhỏ nhất trong Kubernetes. Mỗi Pod có IP riêng và có thể chứa một hoặc nhiều container chia sẻ network namespace và storage.
🌐 Buổi 04 — Networking & Service
0/5 đúng
CÂU 06
Tại sao Pod IP không phải là địa chỉ ổn định để ứng dụng khác phụ thuộc lâu dài?
Khi Pod restart, scale hoặc rolling update, Pod mới sẽ nhận IP khác. Vì vậy ứng dụng không nên hard-code Pod IP mà phải dùng Service để có địa chỉ ổn định.
CÂU 07
Sự khác nhau giữa port và targetPort trong Service spec là gì?
port là port Service lắng nghe (ClusterIP:port). targetPort là port thực tế traffic được forward đến container. Hai giá trị này thường khác nhau.
CÂU 08
Headless Service được tạo bằng cách nào?
Headless Service được tạo bằng cách đặt clusterIP: None. DNS sẽ trả về danh sách IP của các Pod thay vì một ClusterIP ảo.
CÂU 09
Khi EndpointSlice của một Service rỗng, nguyên nhân thường gặp nhất là gì?
Khi selector không khớp label của Pod (typo, namespace sai, label chưa có), EndpointSlice sẽ rỗng và Service không forward traffic đến Pod nào.
CÂU 10
NodePort service có thể được truy cập từ đâu?
NodePort mở một port (30000-32767) trên TẤT CẢ các node. Bạn có thể truy cập qua bất kỳ NodeIP:nodePort nào — kube-proxy sẽ forward đến đúng Pod.

# Xem các pod hệ thống
kubectl get pods -n kube-system

# Xem tất cả namespace
kubectl get namespaces

# Xem tất cả resource trong namespace default
kubectl get all -n default

# Xem cluster info
kubectl cluster-info

# Xem component status
kubectl get componentstatuses

📦 Quản lý Pod

bash
# Xem pod với IP và node
kubectl get pods -o wide

# Mô tả chi tiết pod (xem Events!)
kubectl describe pod <pod-name>

# Xem logs
kubectl logs <pod-name>
kubectl logs <pod-name> -f  # follow

# Exec vào pod
kubectl exec -it <pod-name> -- bash

# Tạo pod tạm thời để test
kubectl run test --image=busybox:1.36 --rm -it -- sh

# Xóa pod
kubectl delete pod <pod-name>

🚢 Quản lý Deployment

bash
# Tạo deployment
kubectl apply -f deployment.yaml

# Scale replicas
kubectl scale deployment <name> --replicas=5

# Update image (rolling update)
kubectl set image deployment/<name> nginx=nginx:1.28

# Xem rollout status
kubectl rollout status deployment/<name>

# Xem rollout history
kubectl rollout history deployment/<name>

# Rollback về version trước
kubectl rollout undo deployment/<name>

# Rollback về revision cụ thể
kubectl rollout undo deployment/<name> --to-revision=2

🌐 Quản lý Service

bash
# Xem service
kubectl get service

# Xem endpoint slices
kubectl get endpointslices

# Kiểm tra DNS từ trong cluster
kubectl run dns-test --image=busybox:1.36 --rm -it -- \
  nslookup <service-name>.<namespace>.svc.cluster.local

# Test kết nối đến service
kubectl run curl-test --image=curlimages/curl --rm -it -- \
  curl http://<service-name>:<port>

# Port-forward để test local
kubectl port-forward service/<name> 8080:80

🐛 Debug và troubleshooting

bash
# Xem events trong namespace
kubectl get events --sort-by=.lastTimestamp

# Kiểm tra resource usage
kubectl top pods
kubectl top nodes

# Kiểm tra node có issue
kubectl describe node <node-name>

# Xem log kubelet trên node
journalctl -u kubelet -f

# Kiểm tra container trực tiếp qua CRI
crictl ps
crictl logs <container-id>

# Debug pod với ephemeral container
kubectl debug -it <pod-name> --image=busybox --target=<container>
